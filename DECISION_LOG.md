## Решение #1: Long Polling вместо Webhook

**Дата:** 2026-03-28  
**Статус:** Принято  
**Автор:** Команда разработки

### Контекст
Необходимо выбрать способ получения обновлений от Telegram Bot API.

### Варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Webhook** | Масштабируемость, низкие задержки | Требует публичный URL, HTTPS, сложнее настройка |
| **Long Polling** | Простота, не требует публичный URL | Задержки, менее масштабируемо, один экземпляр бота |

### Решение
**Выбрано:** Long Polling (GetUpdatesChan, таймаут 60 сек)

### Обоснование
- MVP для разработки и тестирования
- Не требуется публичный URL
- Проще локальный запуск через Docker

### Последствия
- ⚠️ Ограничение на масштабирование (один экземпляр бота)
- ⚠️ При переходе в production потребуется webhook

### Когда пересмотреть
При подготовке к production-развёртыванию с высокой нагрузкой.

## Решение #2: YAML-маппинг Telegram alias в пользователей Todoist + AI-выбор исполнителя

**Дата:** 2026-05-24  
**Статус:** Принято  
**Автор:** Команда разработки

### Контекст
Нужно автоматически назначать исполнителя в Todoist при создании задачи из Telegram-обсуждения.

Ограничения:
- Todoist позволяет назначать задачу только пользователю, который реально состоит в выбранном проекте
- в Telegram люди могут упоминаться по `@username`, имени, фамилии или в свободной фразе вроде `я сделаю` / `пусть возьмёт Денис`
- простое детерминированное сопоставление по тегам или подстроке даёт много ложных назначений

### Варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Полностью ручной выбор исполнителя в preview** | Максимально предсказуемо | Лишние действия для пользователя, слабый UX для MVP |
| **Детерминированный матч по `@mention` / alias** | Простая реализация, быстро работает | Частые ложные срабатывания: людей тегают для внимания, а не для назначения |
| **AI без явного маппинга** | Гибкость в интерпретации текста | Нельзя безопасно назначить задачу без знания допустимых пользователей Todoist |
| **YAML-маппинг + AI среди допустимых кандидатов** | Контролируемый список исполнителей, гибкая интерпретация естественного языка | Требует предварительной загрузки маппинга и поддержки отдельного flow |

### Решение
**Выбрано:** хранить для каждой пары `чат + Todoist-проект` YAML-маппинг `Telegram alias -> Todoist user`, а финальный выбор исполнителя делать через AI только среди кандидатов из этого маппинга.

### Обоснование
- Исключаем назначения на пользователей, которых нет в Todoist-проекте
- Сохраняем удобство естественного языка: AI может интерпретировать фразы вроде `Давайте я сделаю`, `исполнителем должен быть Денис`, `пусть возьмёт Егор`
- Избегаем хрупкой бизнес-логики на жёстких эвристиках по `@mention` и подстрокам
- Делаем конфигурацию прозрачной для команды: маппинг можно обновить YAML-файлом без изменения кода

### Последствия
- Появляется новая команда `/set_assignee_map` и отдельный flow загрузки YAML-файла
- Для каждого проекта нужно поддерживать актуальный маппинг alias
- Качество выбора assignee зависит от качества prompt-а и формулировки обсуждения
- `@mention` сам по себе больше не считается автоматическим назначением

### Когда пересмотреть
- Если появится необходимость массово управлять маппингами через UI, а не через YAML
- Если понадобится более строгий enterprise-flow с обязательным ручным подтверждением исполнителя
- Если AI-качество окажется недостаточным и потребуется гибридный подход с explainable rules
