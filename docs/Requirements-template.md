## Нефункциональные требования (NFR)

> Как система должна работать. Обязательно с числами.

### Производительность

| Метрика | Требование | Как измеряем |
|---|---|---|
| Latency (p99) | ≤ 200ms | APM / Grafana |
| Latency (p50) | ≤ 50ms | APM / Grafana |
| Пиковый RPS | ≥ 500 | Нагрузочное тестирование |
| Время деградации | Не более 1 сек при 2x нагрузке | Стресс-тест |

### Надёжность

| Метрика | Требование |
|---|---|
| Доступность | ≥ 99.9% (не более 8.7 часов простоя в год) |
| MTTR (среднее время восстановления) | ≤ 30 минут |
| RPO (допустимая потеря данных) | 0 (нет потери транзакций) |
| RTO (время восстановления после сбоя) | ≤ 15 минут |

### Масштабируемость

- Горизонтальное масштабирование: сервис должен поддерживать добавление инстансов без изменения конфигурации
- Объём данных: рассчитан на рост до _N_ записей в год

### Безопасность

- Все данные передаются по HTTPS/TLS 1.2+
- ПДн хранятся зашифрованными
- Токены не попадают в логи

### Совместимость

- API обратно совместим в рамках одной major-версии
- _Поддерживаемые клиенты/версии_

## Сценарии использования (Use Cases)

### UC-01: [Название сценария]

**Актор:** Мерчант

**Предусловие:** Мерчант авторизован в личном кабинете

**Основной поток:**
1. Мерчант открывает раздел «Акции»
2. Нажимает «Создать акцию»
3. Заполняет форму: название, скидка (%), период
4. Сохраняет → система создаёт акцию со статусом «Черновик»
5. Мерчант публикует акцию → статус меняется на «Активна»

**Альтернативный поток (ошибка валидации):**
3а. Скидка указана > 100% → система показывает ошибку «Скидка не может превышать 100%»

**Постусловие:** Акция создана и сохранена в системе
